# Phase 0 개발 노트 — 기반 공사 (환경 · 데이터베이스 · 로그인 · 디자인 시스템)

> **이 문서의 목적**: 비개발자(기획자)도 "Phase 0에서 무엇을 깔았고, 화면 뒤에서 어떤 원리로 동작하는지"를 개발 지식과 함께 이해할 수 있도록 쉽게 풀어 쓴 노트입니다. 전문 용어 옆에는 항상 일상 비유를 답니다.

## 0. 한 문장 요약

Phase 1(타이핑 학습)이 이미 동작하고 있었는데, Phase 0은 그 위에 **"여러 사람이·여러 기기에서·인터넷이 끊겨도 안전하게 쓸 수 있는 진짜 서비스"** 가 되도록 **토대(데이터베이스·로그인·동기화·디자인 시스템)** 를 완성하는 단계입니다.

집으로 비유하면 — Phase 1은 "방 한 칸을 예쁘게 꾸민 것", Phase 0은 그 방을 떠받치는 **상하수도·전기·현관문(보안)·내부 인테리어 규격**을 까는 공사입니다.

## 1. 가장 중요한 설계 결정 — "있으면 켜지고, 없으면 조용히 로컬로"

Phase 0의 핵심 원칙은 프로젝트 규칙 6번 **"학습은 절대 멈추지 않는다"** 입니다. 그래서 이렇게 설계했습니다:

- **Supabase(클라우드 DB) 설정값이 있으면** → 구글 로그인 + 기기 간 동기화가 **자동으로 켜짐**.
- **설정값이 비어 있으면** → 앱이 **"로컬 게스트 모드"** 로 동작 → 로그인 없이도 Phase 1 학습이 그대로 됨.

> 💡 **왜 이렇게?** — 개발/시연 중에는 클라우드 연결 없이도 바로 학습 화면을 볼 수 있어야 하고, 인터넷이 끊겨도 학습이 멈추면 안 되기 때문입니다. "연결되면 좋고, 안 되면 혼자서도 잘 도는" 구조를 **graceful degradation(우아한 강등)** 이라고 부릅니다. — 자동문이 고장 나도 손으로 밀고 들어갈 수 있게 만드는 것과 같습니다.

**현재 상태**: `.env.local`에 진짜 Supabase 값이 채워져 있어 앱은 **로그인-퍼스트 모드**로 동작합니다(단어 900행 적재·구글 로그인 배선 완료, 7장 참조). 만약 이 값을 비우면 즉시 게스트 모드로 되돌아가 로그인 없이도 학습이 됩니다 — 이 "양쪽 다 되는" 구조가 graceful degradation의 핵심입니다.

## 3. [0-2] 데이터베이스 — 단어와 학습 기록을 어디에 저장하나

### Supabase란?

**Supabase = 클라우드에 있는 데이터 창고(PostgreSQL 데이터베이스) + 로그인 기능 + 보안 규칙** 을 한 번에 주는 서비스입니다. 우리 앱은 이걸 **"단 하나의 진짜 저장소(single source of truth)"** 로 씁니다.

### 만든 4개의 표(테이블)

`supabase/schema.sql` 파일에 4개의 "표" 설계도를 담았습니다 (엑셀 시트 4장이라고 생각하면 됩니다):

| 표 이름 | 무엇을 담나 | 비유 |
|---------|-------------|------|
| `words` | 영어 단어·예문 900개 (콘텐츠) | 공용 교과서 |
| `user_state` | 유저 1명의 레벨·연속학습일·누적 단어 수·선호 카테고리 | 내 학습 신상카드 |
| `progress` | 단어 하나하나의 학습 신호(맞췄나/복습중인가/언제 다시 볼까) | 단어별 성적표 |
| `study_sessions` | 한 세트(10문제) 끝낼 때마다 요약 1줄 | 학습 일지 (통계·리포트의 원천) |

### 보안의 핵심 — RLS (Row Level Security, 행 단위 보안)

여기가 Phase 0에서 가장 중요한 안전장치입니다.

- **`words`(교과서)** → **"누구나 읽기만 가능, 쓰기 금지"**. 단어는 공개 콘텐츠라 누가 봐도 되지만, 아무도 함부로 고칠 수 없게 막았습니다.
- **`user_state`·`progress`·`study_sessions`(내 학습 데이터)** → **"본인 것만 읽고 쓰기 가능"**. 데이터베이스가 `auth.uid() = user_id`(접속한 사람 = 그 행의 주인)인지 **자동으로 검사**해서, 남의 학습 기록은 아예 보이지도 않습니다.

> 💡 **RLS를 쉽게 말하면** — 도서관(words)은 누구나 책을 읽지만 낙서는 금지, 개인 사물함(progress)은 **내 열쇠로만** 열리는 것과 같습니다. 이 검사를 앱 코드가 아니라 **데이터베이스 자체가** 하기 때문에, 설령 앱에 빈틈이 있어도 남의 데이터가 새지 않습니다.

### 세 가지 열쇠 (키)의 차이 — 매우 중요

| 열쇠 | 권한 | 어디에 두나 |
|------|------|-------------|
| **anon 키** | RLS 규칙 안에서만 동작(읽기 위주, 본인 것만 쓰기) | 프론트엔드에 노출 OK |
| **Service Role 키** | RLS를 **무시**하는 만능 마스터키 | **절대 프론트엔드 금지.** 시드 스크립트에서만 |

`src/lib/supabase/client.ts`는 **anon 키만** 사용합니다. 만능 마스터키(Service Role)는 단어 900개를 한 번에 넣는 `scripts/seed.ts`에서만 쓰고, 그 파일은 사용자 컴퓨터에서만 실행됩니다.

> ⚠️ **왜 이렇게 깐깐한가?** — 만능 마스터키가 웹페이지 소스에 섞여 들어가면, 누구나 그걸 주워 모든 사용자 데이터를 마음대로 고칠 수 있습니다. 그래서 "프론트엔드엔 anon 키만"이 철칙입니다.

### 시드(seed) 스크립트란?

빈 `words` 표에 단어 900개를 **한 번에 부어 넣는** 작은 프로그램입니다(`npm run seed`). 정수기에 물통 끼우듯, DB를 처음 만들 때 딱 한 번 실행합니다.

## 4. [0-3] 구글 로그인 — 현관문 만들기

### 사용자가 보는 것

- 로그인 화면(`/login`)에 큰 **[Google로 계속하기]** 버튼 하나.
- 누르면 → 구글 로그인 창 → 끝나면 다시 우리 앱으로 → 학습 화면으로.
- "같은 구글 계정이면 기기 간 학습이 이어져요 / 이메일·이름·사진만 사용" 안내 한 줄(맥락적 프라이버시 표시, 규칙 6·T9).

### 화면 뒤 원리

- **`AuthProvider`** = 앱 전체를 감싸는 "보안 데스크". 지금 누가 로그인했는지를 항상 알고 있고, 로그인 상태가 바뀌면(`onAuthStateChange`) 즉시 감지합니다.
- **신규 유저가 처음 로그인하면** → `user_state` 표에 그 사람의 기본 행(레벨1·연속0)을 1줄 자동 생성(`ensureUserState`). 이미 있으면 덮어쓰지 않습니다.
- **`/auth/callback`** = 구글에서 돌아올 때 잠깐 들르는 "통관 게이트". 로그인 결과를 정리하고 학습 화면으로 보냅니다.
- **`AuthGuard`(세션 가드)** = 학습 화면 앞을 지키는 문지기. **단, Supabase가 설정된 경우에만** 비로그인 사용자를 로그인 화면으로 돌려보냅니다. 설정이 없으면 문을 활짝 열어 둡니다(게스트 모드).

> 💡 **"OAuth"가 뭔가요?** — 우리 앱이 비밀번호를 직접 받지 않고, "구글이 보증해 주는 신분증"으로 로그인하는 방식입니다. 비밀번호를 우리가 보관하지 않으니 더 안전하고, 사용자는 클릭 한 번으로 끝납니다.

## 5. [0-4] 데이터 레이어 — 읽고·쓰고·끊겨도 안 잃어버리기

이 부분은 "데이터가 오가는 수도 배관"입니다. 규칙 6의 **"키 입력마다 서버를 때리지 않는다 / 오프라인이어도 안 끊긴다"** 를 코드로 구현했습니다.

### (1) 단어 불러오기 — 3단 폴백 (`getWords`)

단어를 가져올 때 **3중 안전망**을 둡니다:
1. **Supabase에서 가져오기** (설정돼 있으면) → 성공하면 그 결과를 브라우저에 **캐시(임시 저장)**.
2. 실패하거나 미설정이면 → **캐시(LocalStorage)** 에서 가져오기.
3. 그것도 없으면 → 앱에 **내장된 words.json**(900개)에서 가져오기.

> 💡 어떤 상황에서도 단어가 안 나오는 일이 없습니다. 인터넷이 끊겨도, 클라우드가 죽어도 학습은 계속됩니다.

### (2) 학습 기록 저장 — "모았다가 한 번에" (`userData.ts`)

- 학습 중에는 키 입력·문제마다 서버에 쓰지 **않습니다**. (그러면 느리고 비싸니까요.)
- 대신 **세션이 끝나는 시점**에 바뀐 부분만 **모아서 한 번에** 저장(`upsert`)합니다.
- `upsert` = "있으면 갱신, 없으면 새로 추가". `progress`는 (유저+단어) 짝을 기준으로 중복 없이 처리합니다.

> 💡 **비유** — 메모를 할 때마다 우체국에 달려가지 않고, 하루치를 모아 한 번에 부치는 것과 같습니다. 빠르고, 데이터 요금도 아낍니다.

### (3) 오프라인 큐 — 인터넷이 끊겼을 때 (`offlineQueue.ts`)

- 저장하려는데 인터넷이 끊겨 있으면 → 그 내용을 브라우저 안 **"보낼 편지함(큐)"** 에 차곡차곡 쌓아 둡니다.
- 인터넷이 다시 연결되면(`online` 이벤트) → **자동으로 밀린 편지를 전부 발송**(flush)합니다.
- 결과적으로 데이터 한 줄도 잃지 않으면서, 학습 흐름은 한순간도 멈추지 않습니다.

### (4) 데이터 설계도 (타입)

`UserState`, `Progress`, `StudySession`의 "정확한 모양"을 TypeScript 타입으로 정의했습니다. 덕분에 잘못된 모양의 데이터를 넣으려 하면 **개발 단계에서 미리 오류**로 잡힙니다(`any` 금지 규칙 준수).

## 6. [0-5] 디자인 시스템 — 앱 전체가 공유하는 부품 상자

매번 버튼을 새로 그리지 않도록, **재사용 가능한 UI 부품**을 한곳에 만들었습니다. 레고 블록 상자라고 보면 됩니다.

### 만든 공통 부품

| 부품 | 설명 | 들어간 디테일 |

| `Button` | 크고 둥근 버튼(3종: 강조/보조/투명) | 누를 때 살짝 눌리는 느낌(scale), 포커스 링, 48px+ |
| `Card` | 둥근 카드 컨테이너 | 일관된 여백·테두리 |

| `Hearts` | 하트 3개(남은 기회) | 줄어들 때 톡 사라지는 모션 |
| `LoadingDots` | 점 3개 로딩 표시 | 통통 튀는 모션 |
| `BottomActionBar` | 화면 하단 고정 버튼 영역 | 홈 인디케이터와 안 겹치게 세이프에어리어 |
| `BottomSheet` | 아래에서 올라오는 시트 | 드래그/바깥탭으로 닫기, **떠 있는 레이어에만 블러**(절제된 글래스, T7) |
| `Toast` | 위에서 잠깐 뜨는 알림 | 성공/정보/주의 3색 + 아이콘 |

### 다크 모드 (T4) — 처음부터 기본 기능

- **`ThemeProvider`** 가 테마를 관리: 사용자가 고른 값(라이트/다크/시스템 따름)을 기억하고, 시스템 설정 변화도 자동 감지.
- **`ThemeToggle`**(해/달 아이콘) 버튼으로 수동 전환. 홈·학습·샘플 화면에 배치.
- **화면 깜빡임(FOUC) 방지**: 페이지가 그려지기 직전에 작은 스크립트가 먼저 테마를 적용 → 새로고침 시 "흰 화면이 번쩍" 하는 현상을 없앴습니다.
- 다크 배경은 순수 검정 대신 **다크 슬레이트(#0F172A 계열)**, 대비는 WCAG AA(4.5:1) 이상.

> 💡 실제로 라이트/다크 양쪽 모두 브라우저에서 캡처로 확인했습니다. 버튼·하트·진행도·카드 모두 양쪽에서 또렷하게 보입니다.

### 접근성 (T5) — 선택이 아닌 기본

- **포커스 링**: 키보드(Tab)로 이동할 때 지금 어디 있는지 테두리로 표시.
- **48×48px 이상 터치 타깃**: 손가락으로 누르기 편한 크기.
- **`aria-label`/`aria-live`**: 화면을 못 보는 분을 위해 스크린리더가 "남은 기회 3개", "정답이에요" 등을 읽어 줌.
- **색만으로 정보 전달 금지**: 정답은 초록색 + **체크 아이콘**을 함께 표시(색맹이신 분도 구분 가능).

### 움직임 (모션) 프리셋

- 슬라이드/페이드 전환을 미리 정의(`lib/motion.ts`)해 화면마다 일관되게 사용.
- **`prefers-reduced-motion` 존중**: 사용자가 기기에서 "동작 줄이기"를 켜 두었다면(멀미·전정기관 민감 등) → `MotionConfig`가 **앱 전체 애니메이션을 자동으로 약하게** 합니다.

### 샘플 페이지 (`/sample`)

위 부품들을 한 화면에 모아 눈으로 확인하는 "쇼룸"입니다. 버튼·진행도·하트·로딩·슬라이드·바텀시트·토스트·다크 토글을 직접 눌러 볼 수 있습니다.

## 7. Supabase 실제 연결 — ✅ 완료됨

아래 5단계가 **모두 완료**되어, 앱은 현재 **로그인-퍼스트 + 기기 간 동기화 모드**로 동작합니다.

| 단계 | 내용 | 상태 |
|------|------|------|
| 1 | Supabase 프로젝트 생성 | ✅ |
| 2 | SQL Editor에서 `supabase/schema.sql` 실행 (표 4개 + RLS) | ✅ |
| 3 | `.env.local`에 URL·anon 키·Service Role 키 입력 | ✅ |
| 4 | `npm run seed` → **words 표에 900행 적재** | ✅ |
| 5 | Google OAuth Provider 활성화 + 클라이언트 ID/시크릿 + 리다이렉트 허용목록 | ✅ |

### 실제로 검증한 것 (실데이터 기준)
- **읽기/쓰기 경계**: anon 키로 단어 읽기 OK / 쓰기(INSERT)는 `42501`로 **차단** / 비로그인 상태 `progress`는 **0행**(RLS 정상).
- **OAuth 배선**: Supabase authorize → `accounts.google.com`으로 **302 리다이렉트**, scope `email profile`, 우리 구글 client_id 정상 포함, 콜백 주소 일치.
- **세션 가드**: 비로그인 상태로 `/study` 진입 시 `/login`으로 자동 이동.

### 구글 OAuth에 등록한 주소 (참고)
- **승인된 리디렉션 URI(구글 측)**: `https://<프로젝트>.supabase.co/auth/v1/callback` ← 구글은 항상 여기로만 돌아옴
- **Supabase Redirect URLs(허용목록)**: `http://localhost:3000/auth/callback` + `http://localhost:3000/**`, Site URL `http://localhost:3000`

> ⏳ **딱 하나 남은 것(검증)**: 실제 "구글 계정 선택 → 동의"는 사람이 직접 해야 합니다. 사용자 브라우저에서 `http://localhost:3000` → [Google로 계속하기]를 한 번 누르면 로그인되고, 신규 계정이면 `user_state`에 기본 행이 자동 생성됩니다. (자동 테스트 브라우저는 구글이 차단해서 이 한 번만 수동입니다.)
>
> 🚀 **배포할 때**: 운영 도메인이 생기면 그 도메인의 콜백 주소를 구글 승인 리디렉션과 Supabase Redirect URLs에 똑같이 추가하면 됩니다.

## 10. 직접 실행해 보는 법

```bash
npm install      # 도구 설치
npm run dev      # http://localhost:3000 (현재 로그인-퍼스트 모드)
npm run build    # 배포용 빌드 + 타입/lint 자동 점검 (현재 통과)
npm run seed     # 단어 900개 적재 (이미 1회 적재 완료, 재실행 시 덮어쓰기)
```
주요 화면: `/login`(로그인) · `/`(홈) · `/study`(학습) · `/sample`(디자인 시스템).
> Supabase가 연결돼 있어 비로그인 시 `/study`는 자동으로 `/login`으로 이동합니다.
