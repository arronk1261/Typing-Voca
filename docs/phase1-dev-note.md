# Phase 1 개발 노트 — 코어 타이핑 퀴즈

> **이 문서의 목적**: 비개발자(기획자)도 "지금 무엇이 만들어졌고, 화면 뒤에서 어떤 원리로 동작하는지"를 개발 지식과 함께 이해할 수 있도록 쉽게 풀어 쓴 노트입니다. 전문 용어가 나오면 바로 옆에 일상 비유로 설명합니다.

---

## 1. 한눈에 보기 — Phase 1에서 무엇을 만들었나

학습의 심장인 **"한글 뜻을 보고 영어 빈칸을 타이핑으로 채우는"** 핵심 루프를 완성했습니다.

사용자가 보는 흐름은 이렇습니다:

1. 첫 화면에서 **[오늘의 10단어 학습]** 버튼을 누른다.
2. 문제 카드가 뜬다 → 위에는 한글 뜻과 한글 문장, 아래에는 빈칸이 뚫린 영어 문장.
   예) `Is the station ___ from here?` / 뜻: "먼" / "여기서 역이 먼가요?"
3. 빈칸 자리에 글자 수만큼 **밑줄(\_ \_ \_)** 이 보인다 → 답이 몇 글자인지 힌트.
4. 타이핑하면 밑줄 위로 글자가 **톡톡 채워진다.**
5. **[확인하기]** 를 누른다.
   - **정답** → 화면이 순간 초록빛으로 번쩍 + 색종이(컨페티)가 터지고 → 그 문장을 따라 말하는 "섀도잉" 자리로 넘어간다 (지금은 안내 화면, 실제 발음 기능은 Phase 3).
   - **오답** → 입력칸이 좌우로 흔들리고(shake), 위쪽 하트가 하나 줄고, "거의 다 왔어요! 한 번 더 볼까요?" 격려 메시지 + 첫 글자 힌트가 뜬다.
6. 이렇게 **10문제**를 풀면 → 결과 화면(학습 단어 목록, 첫 시도 정답 수, 복습 노트 수) + 축하 색종이.

이 모든 것이 **휴대폰 세로 화면(폭 390px)** 에서 매끄럽게 동작하는 것을 실제로 확인했습니다.

## 2. 프로젝트의 뼈대 — 집을 짓기 전 터 닦기

Phase 1을 만들려면 먼저 "개발 환경"이라는 토대가 필요했습니다. 빈 땅이었기 때문에 집(앱)을 올릴 기초 공사를 먼저 했습니다.

### 사용한 핵심 도구들 (왜 썼는지)

| 도구 | 쉬운 설명 | 왜 이걸 썼나 |
|------|-----------|--------------|
| **Next.js** | 웹사이트를 만드는 "공장 설비" 전체 세트 | 화면 이동·페이지 구성을 자동으로 처리. 프로젝트 표준 스택 |
| **React** | 화면을 "레고 블록(컴포넌트)"처럼 조립하는 방식 | 버튼·카드 같은 조각을 만들어 재사용 |
| **TypeScript** | 자바스크립트에 "오타·실수 방지 안전장치"를 단 것 | 데이터 모양이 안 맞으면 미리 경고 → 버그 예방 |
| **Tailwind v4** | 미리 만들어진 디자인 스티커 모음 | CSS를 길게 안 쓰고 빠르게 일관된 디자인 |
| **Zustand** | 앱의 "공용 메모장(현재 상태 저장소)" | 지금 몇 번 문제인지, 하트 몇 개인지를 한곳에 보관 |
| **framer-motion** | 움직임(애니메이션) 전담 도구 | 슬라이드 전환, 글자 채움, 흔들림 등 |
| **lucide-react** | 아이콘 꾸러미 | 하트·마이크·트로피 아이콘 |
| **canvas-confetti** | 색종이 효과 | 정답·완주 시 축하 |

### 폴더 구조 (서랍 정리)

```
src/
├─ app/          → 실제 화면(페이지). 홈(/)과 학습(/study)
├─ components/   → 재사용 조각. ui(공용)와 study(학습 전용)로 분리
├─ stores/       → 공용 메모장(Zustand 상태)
├─ hooks/        → 재사용 기능 묶음 (예: 햅틱 진동)
├─ lib/          → 도구 함수 (단어 불러오기, 정답 비교 등)
├─ types/        → 데이터의 "설계도"(모양 정의)
└─ data/         → 단어 900개가 든 words.json
```

> 💡 **"왜 폴더를 나누나요?"** — 옷장을 상의·하의·양말 칸으로 나누면 찾기 쉬운 것과 같습니다. 나중에 기능이 100개가 되어도 어디에 무엇이 있는지 헷갈리지 않게 하려는 정리정돈입니다.

### 디자인 토큰 (브랜드 색·모양 규칙집)

`globals.css`라는 파일에 앱 전체가 공유하는 **색·둥근 정도·글꼴**을 미리 등록했습니다.
- 메인 색 = 인디고(보라빛 파랑, 집중+신뢰), 포인트 = 앰버(주황, Streak·격려)
- 정답 = 초록, 별점 = 골드, 오답 흔들림 = 절제된 산호색
- 글꼴 3종: 한글/UI용, 영어 문장용, **타이핑 빈칸 정렬용 고정폭 글꼴**(밑줄이 가지런히 정렬되도록)

> 💡 **"토큰"이란?** — "우리 앱 빨강은 이 빨강!"처럼 색·크기에 이름표를 붙여 한곳에서 관리하는 방식입니다. 나중에 "메인 색을 바꾸자" 하면 한 줄만 고치면 전체가 바뀝니다.

## 3. 화면 뒤에서 벌어지는 일 — 단계별 원리

### (1-1) 학습 세션의 골격 만들기

화면을 위·중간·아래 3층으로 나눴습니다.
- **위층(정보)**: 진행 막대(`ProgressBar`)와 하트(`Hearts`) — "지금 몇 번째인지, 기회가 몇 번 남았는지"만 보여주는 정보 영역.
- **중간층(주인공)**: 문제 카드 — 화면의 주인공.
- **아래층(행동)**: `BottomActionBar` — [확인하기]·[다음 문제] 버튼. **엄지손가락이 자연스럽게 닿는 화면 맨 아래**에 고정.

문제는 한 세트에 **10개**. 시작할 때 `getWords`라는 함수가 단어 900개 중 레벨 1짜리를 골라 **무작위로 섞어 10개**를 뽑아 "대기 줄(큐)"을 만듭니다. 같은 문제가 중복되지 않습니다.

문제를 넘길 때는 화면이 **좌우로 슬라이드**됩니다 (현재 문제는 왼쪽으로 빠지고, 다음 문제가 오른쪽에서 들어옴). 종이를 넘기는 느낌을 주려는 의도입니다.

> 💡 **"큐(Queue)"란?** — 놀이공원 줄서기와 같습니다. 먼저 들어온 문제가 먼저 출제되고, 하나 끝나면 다음 사람(문제)이 앞으로 옵니다.

### (1-2) 빈칸 타이핑 경험 — 이번 Phase의 핵심 디테일

영어 문장 `Is the station ___ from here?`에서 `___` 부분을 답(`far`) 글자 수만큼 **밑줄 칸**으로 바꿔 보여줍니다.

화면 뒤에서는 영리한 장치가 숨어 있습니다:
- 진짜 입력칸(키보드 입력을 받는 칸)은 **투명하게 숨겨** 놓았습니다.
- 그 위에 보기 좋은 **밑줄 칸들**을 겹쳐 그립니다.
- 사용자가 글자를 치면 → 숨은 입력칸이 글자를 받고 → 밑줄 위에 그 글자가 **아래에서 위로 살짝 튀어 오르며 나타납니다.**
- 지금 입력할 자리의 밑줄은 **깜빡이는 커서**처럼 brand 색으로 빛납니다.

또한 모바일에서 흔한 문제 두 가지를 미리 막았습니다:
1. **키보드가 입력칸을 가리는 문제** → 입력칸에 포커스가 잡히면 화면을 자동으로 가운데로 스크롤(`scrollIntoView`).
2. **iOS에서 입력 시 화면이 확대되는 문제** → 입력 글자 크기를 16px 이상으로 고정(애플 기기는 16px 미만이면 자동 확대됨).

정답 비교는 너그럽게 합니다: **대소문자 무시**, 앞뒤 공백 무시, 가운데 여러 칸 띄어쓰기도 하나로 정리해서 비교(`normalizeAnswer`). "Far"라고 쳐도 "far"로 인정됩니다.

> 💡 **"투명 입력칸 + 보이는 밑줄"을 왜 따로 두나요?** — 기본 입력칸은 디자인을 예쁘게 꾸미기 어렵습니다. 그래서 "실제로 글자를 받는 칸"과 "예쁘게 보여주는 칸"을 분리해, 기능은 그대로 두고 보이는 부분만 자유롭게 디자인한 것입니다.

### (1-3) 정답·오답 판정과 "감각" 주기

[확인하기]를 누르면 답을 비교합니다.

**정답일 때** (학습자에게 즉각 보상):
- 화면 전체가 0.3초간 **초록빛으로 번쩍**(플래시).
- **색종이(컨페티)** 가 터짐 (0.5초 안에 — 기다림 없이 즉시 보상).
- 휴대폰이 **아주 살짝 진동**(`navigator.vibrate`, 지원 안 하는 기기는 조용히 무시).
- "정답이에요!" + **체크 아이콘**이 함께 뜸 (색맹이신 분도 알 수 있게 색+아이콘 **이중 표시**).
- 잠깐 뒤 → 그 문장을 따라 말하는 섀도잉 자리로 자동 전환.

**오답일 때** (절대 혼내지 않음 — 긍정 강화):
- 입력칸이 좌우로 **부르르 흔들림**(shake).
- 위쪽 **하트가 하나 줄어듦**(애니메이션과 함께).
- 빨강·"틀렸습니다" 같은 부정 표현 대신 → **"거의 다 왔어요! 한 번 더 볼까요?"** + **첫 글자 힌트**.
- 입력칸을 비우고 다시 시도하게 함.
- 만약 **하트 3개를 다 쓰면** → "이 단어는 복습 노트에 담아둘게요 📒"라며 다그치지 않고, 이 문제를 **복습 후보로 기록**한 뒤 다음 단계로 넘어감.

> 💡 **왜 이렇게까지 부드럽게 하나요?** — 이 앱의 철학은 "실패가 사용자 탓처럼 느껴지면 안 된다"입니다. 틀려도 기분 나쁘지 않아야 내일도 학습하러 옵니다(리텐션). 그래서 색·문구·애니메이션을 모두 "격려" 쪽으로 설계했습니다.

### (1-4) 10문제 한 바퀴 완성

문제마다 결과를 **공용 메모장(Zustand)** 에 차곡차곡 쌓습니다. 한 문제당 기록하는 정보:
- 첫 시도에 맞췄는지 (`firstTryCorrect`)
- 하트를 다 써서 복습 후보가 됐는지 (`heartsDepleted`)
- 몇 번 시도했는지 (`attempts`)

10문제를 다 풀면 자동으로 **결과 화면**으로 이동합니다:
- 트로피 + 축하 색종이
- 통계 3종: 학습 단어 수 / 첫 시도 정답 수 / 복습 노트 수
- 오늘 학습한 단어 목록 (단어·뜻·결과 아이콘)
- **[한 세트 더]** 버튼 → 새 10문제로 다시 시작

> ⚠️ 이 결과 화면은 **임시 버전**입니다. 별점·평균 점수·Streak 갱신·Supabase 저장 등 정식 결과 화면은 **Phase 4**에서 완성됩니다.

## 4. 이 프로젝트의 규칙을 어떻게 지켰나

CLAUDE.md의 6대 절대 규칙을 Phase 1에서 어떻게 반영했는지:

- **규칙 1 (작업 단위 테스트)**: 1-1→1-2→1-3→1-4 순서대로, 각 단계마다 빌드 통과 + 실제 화면에서 동작 확인.
- **규칙 2 (주석 최소화)**: 변수·함수 이름만 봐도 뜻이 통하게 작성. 비자명한 한 줄(예: iOS 자동확대 방지)만 짧게 주석.
- **규칙 3 (순차 진행)**: Phase 0 토대 → Phase 1을 섞지 않고 차례로.
- **규칙 4 (비개발자 브리핑)**: 이 문서가 그 결과물.
- **규칙 5 (모바일 최우선)**: 모든 화면을 390px 기준으로 설계·검증. 주요 버튼은 화면 하단(엄지 영역), 터치 타깃 충분히 큼, 한 화면에 한 가지 행동.
- **규칙 6 (학습 무중단)**: 단어를 못 불러오면 LocalStorage 캐시로 폴백. 섀도잉은 보너스 자리(placeholder)로 두어 타이핑 학습이 절대 막히지 않음.

## 5. 아직 가짜(placeholder)이거나 다음 Phase로 미룬 것

| 항목 | 현재 상태 | 언제 진짜가 되나 |
|------|-----------|------------------|
| 섀도잉(따라 말하기) | 안내 화면만 표시 | **Phase 3** (음성 인식·채점) |
| 결과 화면 | 임시(통계 일부) | **Phase 4** (별점·Streak·저장) |
| 로그인·서버 저장 | 없음 (브라우저 안에서만) | **Phase 0-3 / Phase 2** (구글 로그인·Supabase) |
| 레벨 선택 | 무조건 레벨 1 고정 | **Phase 2** (레벨 테스트) |
| 다크 모드 토글 | 색은 준비됨, 켜는 버튼 없음 | **Phase 4~5** |

> 즉, **Phase 1만으로도 "타이핑으로 단어 외우기"는 완전히 동작하는 단독 앱**입니다. 나머지는 그 위에 얹는 층들입니다.

## 6. 직접 실행해 보는 법

```bash
npm install      # 처음 한 번: 필요한 도구 설치
npm run dev      # 개발 서버 켜기 → 브라우저에서 http://localhost:3000
```
브라우저를 휴대폰 크기로 줄이거나 개발자도구의 모바일 모드(폭 390px)로 보면 의도한 모습이 가장 잘 보입니다.

```bash
npm run build    # 배포용 빌드 + 코드 오류/타입 자동 점검 (현재 통과 상태)
```
